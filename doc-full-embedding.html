📦 Versione Full — Sistema di Embedding

Come tutte le dipendenze di Noesis vengono incorporate in un unico file HTML completamente autosufficiente, eliminando qualsiasi dipendenza da CDN o rete.

Panoramica

Noesis esiste in due varianti di distribuzione che condividono esattamente lo stesso codice applicativo, differendo solo nella modalità di caricamento delle dipendenze esterne:

  • Regular (noesis810.html, 313 KB) — le librerie JS/CSS vengono caricate da CDN jsdelivr al momento dell'apertura
  • Full (noesis810-full.html, 1,7 MB) — tutte le stesse librerie sono incorporate direttamente nel file HTML tramite tre tecniche complementari

Il risultato è un file HTML che non contiene nessun riferimento a URL esterni: può essere aperto da un filesystem locale, da una chiavetta USB, da un server intranet isolato, o da qualsiasi ambiente air-gapped senza alcuna riduzione di funzionalità (esclusa la traduzione browser, che dipende da servizi del browser stesso).

Tecnica 1 — Font come data URI

Bootstrap Icons utilizza un webfont (.woff2) che nella versione Regular viene scaricato da CDN. Nella versione Full il font viene convertito in Base64 e incorporato direttamente nella regola @font-face del CSS:

/* Versione Regular — carica il font dalla rete */
@font-face {
  font-family: "bootstrap-icons";
  src: url("https://cdn.jsdelivr.net/npm/bootstrap-icons@1.11.3/font/fonts/bootstrap-icons.woff2")
       format("woff2");
}

/* Versione Full — font incorporato come data URI */
@font-face {
  font-family: "bootstrap-icons";
  src: url("data:font/woff2;base64,d09GMgABAAAAAf1cAA...")
       format("woff2");
}

Il file .woff2 viene codificato in Base64 e incollato direttamente nel valore url(). Il browser lo decodifica in memoria al momento del parsing del CSS, senza nessuna richiesta HTTP. Il CSS Bootstrap Icons è composto da due parti: la regola @font-face e le classi .bi-* che mappano i code point Unicode — entrambe sono incluse nello stesso blocco <style> inline.

Tecnica 2 — Librerie JS inline

Nella versione Regular, JSZip ed epub.js vengono caricati tramite tag <script src="...">:

<!-- Regular: caricamento da CDN -->
<script src="https://cdn.jsdelivr.net/npm/jszip@3.10.1/dist/jszip.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/epubjs@0.3.93/dist/epub.min.js"></script>

Nella versione Full, il contenuto minificato di ciascuna libreria viene incollato direttamente dentro un tag <script> senza attributo src:

<!-- Full: sorgente JS direttamente inline -->
<script>
/*!
 * JSZip v3.10.1
 * ...
 */
!function(e,t){"object"==typeof exports&&"object"==typeof module
?module.exports=t():"function"==typeof define&&define.amd
?define([],t):...
</script>

<script>
!function(t,e){"object"==typeof exports&&"object"==typeof module
?module.exports=e(require("JSZip")):"function"==typeof define
&&define.amd?define(["JSZip"],e):...
</script>

Il browser esegue questi blocchi esattamente come se avesse scaricato i file, ma senza nessuna richiesta di rete. L'ordine di inserimento è importante: JSZip deve precedere epub.js perché quest'ultimo lo usa come dipendenza (require("JSZip") nella versione UMD).

Tecnica 3 — sn56Source blob (editor popup)

L'editor WYSIWYG di Noesis funziona come una finestra popup separata: quando l'utente apre l'editor, JavaScript costruisce dinamicamente un documento HTML completo e lo apre in un nuovo contesto di navigazione (window.open() + document.write()).

Il sorgente di questo documento popup è memorizzato nel DOM principale come stringa JSON all'interno di un tag non-eseguibile:

<script type="application/json" id="sn56Source">
  "...stringa JSON con l'intero HTML dell'editor..."
</script>

L'attributo type="application/json" impedisce al browser di eseguire il contenuto come JavaScript — viene trattato come dati puri, leggibili da document.getElementById('sn56Source').textContent.

Versione Regular — sn56Source con CDN

Nella versione Regular, l'HTML dell'editor referenzia le proprie dipendenze via CDN:

<link href="https://cdn.jsdelivr.net/npm/summernote@0.9.1/dist/summernote-lite.min.css"
      rel="stylesheet">
<link href="https://cdn.jsdelivr.net/npm/bootstrap-icons@1.11.3/font/bootstrap-icons.css"
      rel="stylesheet">
<script src="https://cdn.jsdelivr.net/npm/jszip@latest/dist/jszip.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/turndown/dist/turndown.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/html-docx-js/dist/html-docx.min.js"></script>
<script src="https://cdn.jsdelivr.net/npm/summernote@0.9.1/dist/summernote-lite.min.js"></script>

Versione Full — sn56Source con dipendenze incorporate

Nella versione Full, l'HTML dell'editor contenuto nel blob sn56Source ha tutte e sei le dipendenze incorporate con le stesse tecniche descritte sopra: i font come data:font/woff2;base64,... nel CSS, i sorgenti JS minificati direttamente in <script> inline. Il blob è una stringa JSON (con escape dei caratteri speciali) che racchiude un file HTML completo e autosufficiente.

Questo è il blocco più pesante del file: circa 916 KB, che diventano ~580 KB nello zip grazie alla compressione deflate applicata ai dati base64.

Dimensioni e compressione

La dimensione del file HTML non compresso è 1,7 MB — circa 5× la versione Regular (313 KB). Tuttavia il file si comprime molto bene:

Formato Regular Full Rapporto compressione Full
HTML non compresso 313 KB 1.700 KB —
ZIP (deflate) 64 KB 591 KB ~65% riduzione

La buona compressibilità del file Full è dovuta alla natura stessa del Base64: i dati binari codificati in Base64 hanno un alfabeto limitato a 64 caratteri, il che produce pattern altamente ripetitivi che gli algoritmi deflate comprimono in modo efficiente. I sorgenti JS minificati si comprimono bene per la stessa ragione (keyword ripetute, strutture sintattiche simili).

Cosa non è offline

Anche nella versione Full esistono funzionalità che richiedono connessione per definizione:

  • Traduzione browser — La traduzione automatica si basa sul servizio integrato del browser (Google Translate, DeepL, ecc.), che contatta server remoti. Questa funzione non è emulabile offline e resta non disponibile senza rete in entrambe le versioni.
  • Link aperti dall'app — I link a risorse esterne presenti nei documenti EPUB (immagini remote, riferimenti bibliografici online) dipendono dalla rete, ma sono contenuti dell'utente, non dipendenze dell'applicazione.

Tutte le funzionalità core — lettura EPUB, highlighting, estrazione capitoli, snapshot, editor WYSIWYG, export PDF/DOCX, libreria, collezioni — funzionano completamente offline.

Come aggiornare le dipendenze

Poiché le dipendenze sono incorporate staticamente, non ricevono aggiornamenti automatici come nella versione Regular. Il processo per aggiornare una dipendenza nella versione Full richiede:

  1. Scaricare il nuovo file minificato (.min.js o .min.css) dalla sorgente ufficiale
  2. Se contiene font: convertire i file .woff2 in Base64 (base64 -w0 file.woff2) e sostituire le stringhe data:font/woff2;base64,... nel CSS
  3. Sostituire il corpo dello <script> o del <style> corrispondente nel file HTML
  4. Per le dipendenze dell'editor: aggiornare le occorrenze corrispondenti nel blob sn56Source, avendo cura del corretto escaping JSON (backslash, virgolette, newline)

Per questo motivo la versione Regular è raccomandata per l'uso quotidiano, mentre la versione Full è destinata a scenari che richiedono autonomia dalla rete o archiviazione a lungo termine.

noesis-full-Deps.zip

Il file noesis-full-Deps.zip distribuito insieme alla versione Full contiene i sorgenti originali delle dipendenze incorporate, nelle stesse versioni usate per costruire noesis810-full.html:

  • I file .min.js delle librerie JS (JSZip, epub.js, Summernote, Turndown, html-docx-js)
  • I file .min.css e i font .woff2 di Bootstrap Icons e Summernote

Questo pacchetto ha due utilità principali:

  • Verifica: permette di confrontare le versioni incorporate con quelle ufficiali per auditing di sicurezza
  • Ricostruzione: fornisce il materiale di partenza per ricostruire o aggiornare il file Full senza dover ri-scaricare le dipendenze dalla rete
